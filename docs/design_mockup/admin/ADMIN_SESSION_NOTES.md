## 1. 확정된 결정사항

### 디자인
| 결정 | 내용 |
|---|---|
| admin 디자인 기반 | Linear 스타일 (사이드바 + 별도 페이지) — Toss 스타일(탭+슬라이드)에서 전환 |
| 포인트 컬러 분리 | **user: 블루 `#3182f6`** / **admin: 인디고 `#1E40AF`** — 의도적 분리 |
| 에디터 형태 | 별도 페이지 (`/admin/new`) — 슬라이드 패널 아님 |
| 본문 입력 | 구조화 폼 기본 (4단/3단) + `layout=free` 토글 시 textarea |

### admin 사이드바 구조 (확정)
```
운영
  ├ 콘텐츠     — 실전 케이스 · AI 트렌드 (+ 가이드·프롬프트는 콘텐츠 타입으로 관리)
  ├ 분석       — 북극성 · 퍼널 · KPI · 콘텐츠별 성과
  ├ 가입자     — 유저 목록 + 행동 분석
  ├ 전자책     — 등록/수정 + 주문 내역 + 발송 관리
  └ 도구 관리  — AI 도구 · 프롬프트 · 가이드 (탭 전환)
소통
  ├ 후보 카드  — "이런 거 어때요" 관리
  └ 의견함     — 유저 의견 열람
```

### North Star & KPI
| 항목 | 정의 |
|---|---|
| **북극성 지표** | 주간 프롬프트 활용 유저 수 (프롬프트 1회+ 복사한 주간 UV) |
| 메인 퍼널 | 방문 → 조회 → 정독(70%+) → 프롬프트 복사 → 저장 |
| KPI 카드 5개 | 방문자(UV) · 가입자 · 프롬프트 복사 · 저장률 · 도움률 |
| 가드레일 | 이탈률 >70% · 부정반응 >20% · 체류 <2분 · 발송실패 >5% · 로그인전환 <30% |

### GA4 연동
| 항목 | 내용 |
|---|---|
| 역할 분리 | 자체 events = admin 대시보드 소스 / GA4 = 유입·퍼널·리텐션 |
| Consent | consent_analytics=false → GA denied (모델링만) / 자체 events는 항상 기록 |
| UTM 규칙 | instagram/newsletter/kakaotalk/brunch 등 채널별 규칙 확정 |
| 스크롤 추적 | 25/50/70/100% 커스텀 (GA4 기본 90%로는 정독률 측정 불가) |

### MVP 범위 확장 (user 시안 반영)
| 추가 항목 | 설명 |
|---|---|
| 가이드 (`/guides`) | 공식 문서 큐레이션 — 자료실 하위 |
| AI 도구 (`/tools`, `/tools/[slug]`) | 추천 도구 목록 + 상세 |
| 프롬프트 모음 (`/prompts`) | 바로 쓰는 프롬프트 — 자료실 하위 |
| 온보딩 (`/onboarding`) | 최초 가입 후 1회 — 직업·직무·관심주제·AI도구 선택 |
| admin 도구 관리 (`/admin/tools`) | 도구·프롬프트·가이드 탭 전환 CRUD |

## 5. 추가 결정 필요한 사항

### 5-1. 콘텐츠 타입 확장 — DB 구조

현재 `contents.content_type`은 `case | trend`만 있음. 가이드·프롬프트를 어떻게 다룰지:

| 옵션 | 장점 | 단점 |
|---|---|---|
| A) `content_type`에 `guide | prompt` 추가 | 기존 contents 테이블 재활용, 검색·필터 통합 | body jsonb 구조가 타입별로 달라짐 |
| B) `tools`, `prompts`, `guides` 별도 테이블 | 각 타입에 맞는 스키마 | 테이블 수 증가, 관리 복잡 |
| C) `resources` 통합 테이블 (도구·가이드용) + 프롬프트는 contents에서 자동 추출 | 절충 | 설계 복잡도 중간 |

→ **결정 필요**: contents 테이블 확장 vs 별도 테이블

### 5-2. 도구 관리 — 데이터 모델

도구(`tools`) 테이블을 신설한다면:

```
tools
  id · slug · name · description · detail_body(jsonb/md)
  · icon_url · category(리서치|분석|제작|자동화)
  · pricing(무료|유료|프리미엄)
  · external_url · is_public · linked_content_ids(uuid[])
  · created_at · updated_at
```

→ **결정 필요**: 이 스키마로 갈지, contents에 통합할지

### 5-3. 프롬프트 모음 — 자동 vs 수동

| 옵션 | 설명 |
|---|---|
| A) 콘텐츠 내 프롬프트 자동 추출 | 콘텐츠 발행 시 body 안의 프롬프트 블록을 `/prompts`에 자동 노출 |
| B) 별도 수동 등록 | admin에서 프롬프트를 독립적으로 등록·관리 |
| C) 둘 다 | 자동 추출 + 운영자가 추가로 등록 가능 |

→ **결정 필요**: 자동 추출 범위와 별도 등록 여부

### 5-4. 온보딩 → 데이터 활용

온보딩에서 수집하는 정보 (user 시안 기준):
- 직업 (단일 선택)
- 직무 (단일 선택)
- 관심 주제 (복수 선택)
- 사용 중인 AI 도구 (복수 선택)

→ **결정 필요**:
- `users` 테이블에 `job_category` 외에 `job_title`, `interests(text[])`, `ai_tools(text[])` 추가?
- 이 데이터를 admin 분석의 어디에 반영할지 (가입자 직무별 분포는 이미 있음, 관심 주제·AI 도구별 분포 추가?)
- 온보딩 완료율을 가드레일에 추가할지

### 5-5. admin 분석 — 유입 경로 패널

현재 유입 경로는 GA4에서만 보기로 했는데:

| 옵션 | 시기 |
|---|---|
| A) MVP는 GA4 콘솔에서 직접 확인 | MVP |
| B) GA4 Data API 연동해서 admin에 유입 패널 표시 | Phase 2 |
| C) 자체 referrer 수집으로 간이 유입 채널 표시 | MVP 가능 |

→ **결정 필요**: 유입 경로를 admin 안에서 보고 싶으면 C 옵션 추가 검토

### 5-6. "풀 설정 페이지" 재검토

spec §8.3에서 비목표로 제외했지만, 온보딩이 추가되면서 계정 페이지에서 관리할 항목이 늘어남:
- 직업·직무·관심주제·AI도구 변경
- 분석 동의 토글
- 프로필 정보

→ **결정 필요**: 계정 페이지를 좀 더 확장할지, 여전히 최소로 유지할지

### 5-7. admin 콘텐츠 폼 — 발행자 필드

이번 세션에서 발행자 필드를 삭제했음 (운영자 1명이므로). 하지만 추후 게스트 기고나 외부 필자를 받을 경우:

→ **결정 필요**: 발행자 필드를 완전히 제거할지, DB에는 남기되 UI에서만 숨길지

### 5-8. user↔admin 컬러 토큰 정리

현재 미세 차이:
| 토큰 | user | admin |
|---|---|---|
| 배경 | `#fff` / `#f7f7f7` | `#FAFAF7` |
| 포인트 | `#3182f6` | `#1E40AF` |
| border | `#e5e8eb` | `#E5E1D5` |

→ **결정 필요**: 배경·border 톤도 의도적 분리인지, 개발 시 하나로 통일할지. 개발 시 디자인 토큰 파일(`tokens.ts`)에서 user/admin 세트를 분리 관리하는 게 좋음.
